📍 为什么 gl_FragCoord.xy / u_resolution.xy 能得到 [0, 1]？

几乎每个 Shader 的第一行都是这句。它不是在「魔法变换坐标」，而是在算一个极其朴素的问题：当前像素在屏幕宽/高方向上，占了多少比例？

一、两个输入各是什么？

变量含义典型范围（以 800×600 为例）
gl_FragCoord.xy 当前像素的门牌号（第几列、第几行），单位是像素 x ∈ [0, 800]，y ∈ [0, 600]
u_resolution.xy 画布总尺寸（宽、高），也是像素数 (800, 600)

GPU 为每个像素并行执行同一段代码。左下角像素拿到 (0, 0)，正中央拿到 (400, 300)，右上角拿到 (800, 600)（具体上界因实现略有差异，但边缘一定对应 0 和 1）。

注意：WebGL 里 gl_FragCoord 的原点在左下角，Y 轴向上——和网页 CSS 的左上角原点相反。

二、核心原理：除法 = 求占比

uv.x = gl_FragCoord.x / u_resolution.x
读作：「当前列号 ÷ 总列数 = 横向走了多少比例」

这和日常算进度条一模一样：

  • 跑了 400 米，全程 800 米 → 400 / 800 = 0.5 → 完成一半
  • 像素在第 0 列 → 0 / 800 = 0.0 → 最左边
  • 像素在第 800 列 → 800 / 800 = 1.0 → 最右边
  • 像素在第 400 列 → 400 / 800 = 0.5 → 正中间

Y 方向完全同理：uv.y = gl_FragCoord.y / u_resolution.y，从底边的 0 到顶边的 1。

所以结果天然落在 [0, 1]：分子是「走了多远」，分母是「总共多远」，比值就是 0%～100% 的小数形式。

三、写成通用映射公式

这其实是最简单的线性重映射，把区间 [0, 分辨率] 映到 [0, 1]：

y = (x − 起点) / (终点 − 起点) × (目标上界 − 目标下界) + 目标下界

// 代入：起点=0，终点=resolution，目标=[0,1]
uv = (gl_FragCoord − 0) / (u_resolution − 0) × (1 − 0) + 0
   = gl_FragCoord / u_resolution

后面课程里的 uv × 2 − 1（映到 [-1,1]）、uv − 0.5（居中）都是同一套公式换不同的起点/终点。详见 坐标居中原理。

五、vec2 除法：x、y 各算各的

vec2 uv = gl_FragCoord.xy / u_resolution.xy;

// 等价于：
// uv.x = gl_FragCoord.x / u_resolution.x;
// uv.y = gl_FragCoord.y / u_resolution.y;

GLSL 对 vec2 / vec2 是分量对应相除，不是某种二维几何运算。横向占比和纵向占比独立计算，互不干扰。

六、为什么要归一化？

分辨率无关

若直接用像素坐标 gl_FragCoord.x，在 1920 宽屏上正中央是 960，在 800 宽屏上却是 400——同一段 Shader 换屏幕就变形。归一化后中央永远是 0.5，左边永远是 0，右边永远是 1。

和颜色范围对齐

Shader 颜色分量也在 [0, 1]。把坐标映到同一区间后，可以直接写 vec3(uv.x, uv.y, 0.5) 把位置变成颜色——第一课渐变就是这么来的。
